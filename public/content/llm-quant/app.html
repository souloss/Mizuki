          💡
          
记忆注入让 LLM "有经验"：没有记忆的 LLM 每次决策都是"第一次"，无法从过去的错误中学习。注入交易记忆后，LLM 能识别连续亏损模式并自动收缩仓位，表现出类似"冷静期"的行为。

📝实际发送给 Agent 的 Prompt (技术分析师)

        

      


      

        💡
        
为什么要多 Agent？单个 LLM 容易产生"确认偏误"——只寻找支持自己判断的证据。多 Agent 架构通过专业分工和交叉验证，显著降低这种偏误。每个 Agent 看到相同数据，但被 Prompt 引导从不同角度分析。

      


      

        
        
      

    


    
    

      

        
Phase 04

        
决策综合阶段

        
Meta Agent（元智能体）接收三个分析结果，综合权衡后输出最终的 JSON 交易决策。这是整个系统的核心——将多维度分析转化为可执行的结构化指令。

        ⚠️
        
JSON 解析是脆弱环节：LLM 可能在 JSON 中夹带解释文本、使用不一致的字段名、或输出无法解析的格式。生产系统必须有重试机制（最多 3 次）和兜底策略（解析失败时 hold 不操作）。

      


      

        
        
      

    


    
    

      

        
Phase 05

        
执行与风控校验阶段

        
LLM 输出的决策必须经过代码层的硬性风控校验，才能被自动执行。即使 Prompt 中声明了风险规则，代码校验仍是最后防线。

      


      

        
        

          
🛡️风控硬校验

          

            

              ⏳
              仓位检查：position_size_pct ≤ 30%
            

            

              ⏳
              止损检查：stop_loss 必须设置且合理
            

            

              ⏳
              单笔风险检查：风险 ≤ 总资金的 2%
            

            

              ⏳
              杠杆检查：10x ≤ leverage ≤ 20x
            

            

              ⏳
              方向检查：direction 必须为 long 或 short
            

          

        


        
        

          
🚀执行日志

          

        

      


      

        📌
        
软约束 vs 硬约束：Prompt 中的风险规则是"软约束"——LLM 可能忽略它们。代码层的校验是"硬约束"——不合规的决策会被自动修正或拒绝。这种双层防护是 LLM 交易系统的标准实践。

      


      

        
        
      

    


    
    

      

        
Phase 06
